# SQLWarden Architecture

Updated: 2026-06-11

SQLWarden is a self-hosted database access platform and SQL IDE. The current repository contains a Go backend, embedded React SPA, custom RBAC engine, database connection/session manager, and workspace file storage foundation. Future product direction includes Wails desktop packaging, SSO/SCIM, stronger audit/compliance features, connector agents, and broader file/storage backends.

This document is the committed architecture source of truth for the repository. When it conflicts with code, migrations, or tests, code wins.

## Current Product Shape

Implemented today:

- One Go server binary in `cmd/api`.
- HTTP application package in `internal/web`, reusable by future entrypoints such as Wails desktop.
- REST API under `/api/v1`.
- Embedded React SPA served from `assets/static`.
- First-run setup through `POST /api/setup`.
- Local username/password auth, JWT access tokens, refresh tokens, database-backed auth sessions, and org access-session revocation.
- Instance-admin layer above organizations.
- Organization, workspace, environment, connection, team, role, policy, and account models.
- Optional personal spaces under `/api/v1/me`.
- Custom additive RBAC enforcer in `internal/access`.
- Effective permissions API and permissions catalog API for frontend capability gating.
- Workspace direct/team membership model and `workspace_members` policy principal.
- Live target database sessions through `internal/connection`.
- Foreground query cancellation through request cancellation.
- Target database drivers for PostgreSQL, MySQL, and SQLite.
- Server-side SQLite target connection gating through `drivers.sqlite.allowed_sources`.
- Workspace file metadata/content APIs for private and shared files.
- Filesystem-backed file content storage under `~/.sqlwarden/files` by default.
- Workspace file content retention reaper.
- DSN and file encryption key rotation foundation.
- Schema introspection abstraction, cache, and API.
- React 19 frontend with TanStack Router, TanStack Query, Tailwind CSS 4, shadcn/ui, Base UI primitives, CodeMirror 6, Zustand, IndexedDB, Y.js, and BroadcastChannel.
- IDE workspace tabs, explorer, file tabs, console tabs, query execution, results pane, editor theme preferences, and same-browser cross-window sync.
- Config through spf13/viper with config file, environment variables, and CLI flags.
- SQLite app database by default at `~/.sqlwarden/sqlwarden.db`; PostgreSQL app database support exists.

Explicitly future or incomplete:

- Wails desktop binary.
- SSO, SCIM, invitations, and enterprise identity lifecycle.
- Multi-backend desktop/server selector UX.
- Connector agent and WebSocket routing for databases behind firewalls.
- Background query runs and query-run observability.
- Full audit log product surface, tamper-evident audit logs, and SIEM forwarding.
- Deny rules and binding expiry enforcement.
- Distributed RBAC cache invalidation.
- Shared-file collaborative editing through WebSockets.
- File uploads, revision browsing UX, S3-compatible file storage, and storage migration tooling.
- Full SQL parser/AST-based query classification.
- PWA service worker setup.

## Configuration

Configuration is loaded by `internal/web` using spf13/viper. Supported sources are config file, environment variables, and CLI flags. Defaults are intended to support local development with minimal setup.

Important concepts:

- `deployment_mode` describes runtime packaging and operating context, such as server or future desktop.
- `access_mode` describes account/authorization behavior, such as multi-user or single-user.
- `personal_spaces_enabled` gates `/api/v1/me/workspaces...`.
- Session revocation can be enabled/disabled for deployments that do not need account session management overhead.
- File storage currently supports local filesystem storage. Config names are designed around active backend plus future backend registry.
- Target SQLite connections are explicitly gated through `drivers.sqlite.allowed_sources`; REST clients cannot rely on the frontend driver list as the security control.

Defaults:

- App DB: `~/.sqlwarden/sqlwarden.db`
- File storage: `~/.sqlwarden/files`

## Request Lifecycle

Typical org-scoped request:

```text
HTTP request
  -> chi route
  -> authenticateV1
  -> requireAccount
  -> orgCtx
  -> workspace/environment/connection context middleware as needed
  -> concrete permission middleware
  -> handler
  -> internal/database or service layer
  -> response.JSON / centralized API error helpers
```

Concrete permission middleware variants are preferred:

- `requireOrgPermission`
- `requireWorkspacePermission`
- `requireEnvironmentPermission`
- `requireConnectionPermission`

The old generic permission middleware was removed to keep route authorization readable.

## API Contract

### Errors

Error responses use a standard envelope:

```json
{
  "error": {
    "code": "validation_failed",
    "message": "Name is required.",
    "field_errors": {
      "name": "Name is required."
    },
    "errors": [],
    "details": {}
  }
}
```

Fields:

- `code`: stable machine-readable error code.
- `message`: user-facing summary.
- `field_errors`: optional map for form fields.
- `errors`: optional non-field validation messages.
- `details`: optional structured metadata.

Common HTTP semantics:

- `400`: malformed JSON or primitive parse errors.
- `401`: unauthenticated or invalid authentication token.
- `403`: authenticated but not permitted.
- `404`: missing resource or resource outside requested scope.
- `409`: state conflict.
- `422`: validation failure.
- `500`: unexpected server error.

### Lists

UI-facing paginated endpoints use:

```json
{
  "items": [],
  "page": 1,
  "page_size": 25,
  "total": 0
}
```

Shared query parameters:

- `page`
- `page_size`
- `sort`
- `order`
- `q`

Resource-specific filters stay as flat query params, for example `driver`, `access_mode`, `role`, `slug`, or `name`.

Small non-paginated list endpoints must still avoid top-level arrays. Workspace file list/recent endpoints return:

```json
{
  "files": []
}
```

### Single Resources And Actions

Many current create/update/get endpoints still return the raw resource directly. This is accepted current state but should be cleaned up incrementally if the API contract is tightened further. Query execution and live database session endpoints have specialized response shapes and should be changed carefully because frontend IDE behavior depends on them.

## Authentication And Sessions

Implemented auth flow:

- First-run setup creates the initial account and instance admin.
- In multi-user mode, setup also requires creation of the first organization.
- In single-user mode, setup seeds a local organization and grants the account owner-level access through normal RBAC.
- Login creates a refresh token, JWT access token, and auth session.
- Refresh-token rotation issues new access tokens.
- Auth sessions and org access sessions are database-backed and revocable when session revocation is enabled.
- Account/session APIs support listing and revoking active sessions.

Current identity model:

- Accounts are global identities.
- Organizations contain memberships in `org_members`.
- Org access is not granted solely by role bindings; org membership is a separate gate.
- Future SSO/SCIM should extend account identity and org access-session semantics rather than replacing the org membership gate.

## Authorization And RBAC

The RBAC implementation lives in `internal/access`.

Core properties:

- Additive allow model.
- Default deny.
- Builtin roles are immutable.
- Role bindings are idempotent.
- Permission checks evaluate the current resource and allowed ancestor resources.
- Policy subjects include accounts, teams, `org_members`, and `workspace_members`.
- Org membership is required before org resource access, regardless of policy bindings.
- Personal spaces use owner middleware and are outside org RBAC.

### Identity And Bootstrap

Accounts are global identities. Account records carry identity and login state, not authorization state. Effective authorization is derived from org membership, workspace membership, teams, roles, and role bindings.

Important identity tables:

- `accounts`: email, name, nullable password for future SSO-only accounts, and active state.
- `instance_admins`: global instance administrators. This is an instance-management layer, not an org permission source.
- `organizations`: org slug/name.
- `org_members`: account membership in an organization.
- `teams` and `team_members`: org-scoped team principals.

`POST /api/setup` is self-sealing. In multi-user server mode it creates the first account, makes that account an instance admin, creates the first organization, adds the account to that organization, and grants the account the builtin `Owner` role through a normal role binding. In single-user mode it still creates a real local account and local organization; it does not bypass RBAC for org-owned resources.

Register/login flows are separate from authorization. An account may exist without meaningful access until it is an org member and receives policy-derived permissions.

### Resource Model

The org resource tree is strict:

```text
organization -> workspace -> environment -> connection
```

`resource_hierarchy` is the source of truth for parent/child ancestry used by RBAC inheritance and discovery. Resource creation helpers must populate hierarchy rows.

Workspace ownership has two modes:

- Org-owned workspace: `org_id = organizations.id`, `owner_type = "org"`, `owner_id = organizations.id`.
- Personal-space workspace: `org_id = NULL`, `owner_type = "space"`, `owner_id = accounts.id`.

Connections are strict leaves under environments. They should not be modeled as dual-parent resources even though the connection row stores both `workspace_id` and `environment_id`; RBAC ancestry is connection -> environment -> workspace -> org.

### Builtin Roles And Default Bindings

Builtin org roles:

- `Owner`: all org, workspace, workspace-file, environment, connection, and policy permissions.
- `Administrator`: all non-owner organization administration permissions. It intentionally excludes `org:delete` and `org:transfer_ownership`.
- `Baseline Access`: baseline org visibility through `org:read`.

Builtin workspace roles:

- `Workspace Admin`: workspace administration, workspace files, environment management, connection management/querying, and workspace policy management.
- `Workspace Member`: baseline workspace visibility through `ws:read`.

Org creation seeds the builtin org roles and a default binding from the special `org_members` principal to `Baseline Access`. That default policy is explicit and revokable; org membership itself does not hard-code `org:read`.

Workspace creation seeds the builtin workspace roles and a default binding from the special `workspace_members` principal to `Workspace Member`. Workspace creation also transactionally creates the default environment. The workspace creator is bound to `Workspace Admin` at the workspace resource.

Permission namespaces:

- `org:*`: organization read/write/delete/invite/ownership-transfer actions.
- `ws:*`: workspace read/write/create/delete actions.
- `wsfile:*`: workspace file read/create/write/delete actions.
- `env:*`: environment read/write/create/delete/deploy actions.
- `conn:*`: connection read/write/create/delete/query actions.
- `policy:*`: role binding and policy read/modify actions.

Query permissions:

- `conn:dql` for read/select-style queries.
- `conn:dml` for data mutation queries.
- `conn:ddl` for schema mutation queries.
- `conn:execute` allows all query classes.

Current query classification is intentionally simple and should be replaced by SQL parser/AST classification later.

### Roles

Roles are org-owned definitions:

- `roles.org_id` always identifies the owning organization.
- `roles.workspace_id IS NULL` means an org-level role definition.
- `roles.workspace_id = N` means a workspace-owned role definition.
- `roles.scope_type` defines the type of resource the role may be bound to.
- `roles.is_builtin` protects seeded roles from mutation/deletion.

Workspace-owned roles can still have different scopes. For example, a workspace-owned connection-scoped role can be bound only to connection resources in that workspace. A workspace-owned environment-scoped role can be bound only to environments in that workspace.

Direct permission bindings are intentionally not part of the current API. Permissions are assigned to roles, and roles are bound to principals at resources.

### Scope Versus Resource Applicability

There are two permission maps and they answer different questions.

Scope validity answers: "Can a role with this `scope_type` contain this permission?"

- Org-scoped role: org, workspace, workspace-file, environment, connection, and policy permissions.
- Workspace-scoped role: workspace, workspace-file, environment, connection, and policy permissions.
- Environment-scoped role: environment and connection permissions.
- Connection-scoped role: connection permissions.

Resource applicability answers: "Is this permission meaningful when checking or displaying capabilities for this target resource?"

This distinction matters. `ws:create` is valid in an org-scoped role because workspaces are created under an org. `ws:delete` is also valid in an org-scoped role and is meaningful when checking a concrete workspace, but it is not valid inside a workspace-scoped role because deleting the workspace itself is a parent-level administrative action.

The backend permissions catalog is the source of truth for:

- permission strings
- labels
- descriptions
- role scope maps
- resource applicability maps

The catalog response includes permission details, scope maps/details, and resource maps/details. The frontend may keep stable permission string constants for checks, but labels, descriptions, role-authoring scope maps, and resource applicability maps must come from the catalog API.

### Policy Bindings

Role bindings are the policy model. A binding connects one role to one principal at one resource.

Binding fields:

- `org_id`: owning organization.
- `role_id`: role definition.
- `subject_type`: `account`, `team`, `org_members`, or `workspace_members`.
- `subject_id`: account ID, team ID, org ID for `org_members`, or workspace ID for `workspace_members`.
- `resource_type`: `org`, `workspace`, `environment`, or `connection`.
- `resource_id`: resource ID of that type.
- `expires_at`: present for future expiry enforcement.
- `created_by`: actor account.

Bindings are unique on `(role_id, subject_type, subject_id, resource_type, resource_id)`, making grant operations idempotent.

`org_members` is a special principal backed by the `org_members` table. It is not a hidden team. `workspace_members` is a special principal backed by direct workspace members plus teams attached to the workspace.

Org policy APIs manage bindings on the current org resource. Workspace policy APIs can bind workspace-owned roles to the workspace, environments, or connections, but the server must verify that selected environment/connection resources belong to the requested workspace.

### Authorization Evaluation

The enforcer checks:

```text
Can(ctx, account_id, org_id, owner_type, resource_type, resource_id, permission)
```

Evaluation flow:

- If `owner_type = "space"`, allow only after `/me` middleware has already proven the current account owns the personal-space resource.
- Resolve principals for the account: direct account principal, org teams, `org_members` when applicable, and `workspace_members` when applicable.
- Resolve ancestry for the target resource from `resource_hierarchy`: target resource, parent resources, and the owning org.
- Load org policy from the process-local cache.
- Match role bindings at the target resource and ancestors.
- Allow if any matching role contains the requested permission and the permission is valid for that role/resource relationship.

Inheritance is additive. Parent bindings can grant access to children. There are no deny rules, so revocation means removing the granting binding or membership source.

### Effective Permissions

`GET /api/v1/orgs/{org_slug}/permissions/effective` exposes effective permissions for UI capability gating.

Parameters:

- `resource_type`: defaults to `org`; accepts `org`, `workspace`, `environment`, and `connection`.
- `resource_id`: required for workspace, environment, and connection.

The API reuses the same enforcer data path as authorization middleware and returns deduplicated, sorted permission strings. It is an advisory UI helper only; handlers and middleware still enforce authorization server-side.

Typical UI usage:

- Fetch once per context page rather than making one API call per button or menu item.
- Check create actions on the parent container: `ws:create` at org, `env:create` at workspace, `conn:create` at workspace/environment.
- Check existing resource actions on the concrete resource: workspace actions at workspace, environment actions at environment, connection actions at connection.

The effective-permissions endpoint validates resource ownership. Missing or cross-org resources should behave as not found. Personal-space permissions are not exposed as org-scoped RBAC.

### Middleware Mapping

Org-scoped routes typically use:

```text
authenticateV1 -> requireAccount -> orgCtx -> concrete resource ctx -> require*Permission -> handler
```

Concrete permission middleware keeps checks readable and prevents accidental parent/child mismatches:

- `requireOrgPermission`: owner type `org`, resource `org`, current org ID.
- `requireWorkspacePermission`: workspace owner type, resource `workspace`, current workspace ID.
- `requireEnvironmentPermission`: workspace owner type, resource `environment`, current environment ID.
- `requireConnectionPermission`: workspace owner type, resource `connection`, current connection ID.

Role-like route guards map to permissions rather than special-case role names:

- Owner-level guard: `org:transfer_ownership`.
- Administrator-level guard: `org:write`.

Personal-space routes use their own `/me` middleware chain:

```text
authenticateV1 -> requireAccount -> requirePersonalSpacesEnabled -> spaceWsCtx/spaceEnvCtx/spaceConnCtx -> handler
```

The `/me` context middleware proves `owner_type = "space"` and `owner_id = current account` before the enforcer owner short-circuit can apply.

### Scope And Discovery Rules

Discovery uses parent/self/child semantics:

- Parent binding check asks: "Does the user have any permission on this parent that would make this child visible?"
- Self binding check asks: "Does the user have any permission directly on this resource?"
- Child binding check asks: "Does the user have any permission on a descendant that requires showing this parent as navigation context?"

For workspace discovery:

- Org/workspace bindings: `ws:*`, `env:*`, or `conn:*`
- Environment bindings: `env:*` or `conn:*`
- Connection bindings: `conn:*`

For environment discovery:

- Org/workspace/environment bindings: `env:*` or `conn:*`
- Connection bindings: `conn:*`

For connection discovery:

- Only `conn:*` applies.

Invalid role scope/resource combinations should be ignored defensively by discovery queries even if earlier API invariants fail.

The discovery queries intentionally check only permissions that make sense for the level being discovered. For example, connection discovery checks `conn:*` only. Workspace discovery can consider child-level environment/connection permissions because a user may need to see a workspace as navigation context to reach a permitted child.

### Caching

RBAC caches are process-local and TTL-based. They improve hot-path request performance but are not distributed cache invalidation.

Cache categories:

- Org policy cache keyed by org ID.
- Principal cache keyed by org ID/account ID.
- Ancestry cache keyed by resource type/resource ID.

Policy changes invalidate org policy cache. Team membership changes invalidate principal cache. Resource deletion invalidates ancestry cache when relevant. A future multi-node deployment needs distributed invalidation or a shorter-lived/shared cache strategy.

### Privileged Org Policy Grants

`policy:modify` permits policy management but does not automatically permit self-promotion to owner-level powers. Org role bindings that grant privileged owner-level permissions such as `org:delete` or `org:transfer_ownership` require the actor to already hold those permissions at that org. This prevents an administrator with policy management from binding themselves to `Owner`.

## Personal Spaces

Personal spaces are implemented under `/api/v1/me`.

Properties:

- Gated by the persisted personal-spaces feature setting.
- `owner_type = "space"` and `owner_id = account.id`.
- Use `/me` owner middleware instead of org RBAC.
- Intended for user-owned personal database workspaces in multi-user deployments.
- Not the desktop/single-user authorization model.

Single-user/local mode should use the seeded local organization and normal org routes for managed resources.

## Workspace Membership

Workspace membership is explicit:

- `workspace_members` stores direct account membership.
- `workspace_teams` stores team membership.
- Effective workspace users can include direct members and inherited team members.
- `workspace_members` is also a special policy principal that can be bound to workspace, environment, or connection resources inside that workspace.

Being a workspace member does not automatically grant environment, connection, query, file, or policy access. The default `workspace_members -> Workspace Member` binding grants baseline `ws:read` only. Additional access must come from explicit policies.

The `workspace_members` principal can be bound to child resources, but only for resources inside that workspace. This allows policies like "all current and future workspace members can read this environment" without turning workspace membership itself into a broad implicit permission.

Cross-org and cross-workspace boundaries must be enforced when creating workspace membership and when binding `workspace_members` to child resources.

## Query Execution And Live Sessions

`internal/driver` is for target databases. It is separate from `internal/database`, which stores SQLWarden metadata.

Implemented target drivers:

- PostgreSQL
- MySQL
- SQLite

`internal/connection` manages live target database sessions:

- Sessions are keyed by account and connection.
- Session IDs are passed through `X-Warden-Session`.
- Queries on a live session are serialized.
- Idle sessions are reaped.
- Foreground query cancellation is supported via request context cancellation.
- Backend session sync lets the IDE show connected state across same-browser windows.

Future phases:

- Background query runs with their own serialized DB session.
- Query-run listing/observability.
- Admin cancellation.
- WebSocket updates for query state.

## Workspace Files

Workspace files are scoped to workspaces and can be:

- private to an account
- shared within a workspace

Authorization differs by visibility:

- Private files require effective workspace membership and ownership of the file.
- Shared files require the relevant `wsfile:*` permission on the workspace.

This logic lives in `internal/files.Service` rather than only in route middleware because the same route family handles private and shared file behavior.

Implemented:

- Metadata in `workspace_files`.
- Content metadata in `workspace_file_contents`.
- Deletion queue in `workspace_file_content_deletions`.
- Filesystem-backed content storage through `internal/filestore`.
- Private/shared file APIs.
- Recent file APIs.
- Browser/tree API.
- Content save/read APIs with ETag support.
- Retention reaper for stale content.
- File encryption key rotation foundation.

Current default storage path is `~/.sqlwarden/files`.

Future:

- Upload arbitrary files through the UI.
- Revision browsing and selective revision retention.
- S3-compatible storage backend.
- Storage backend migration.
- WebSocket-backed shared collaboration.

## Security And Compliance Posture

Current implemented controls:

- Credentials encrypted at rest.
- Key rotation foundation for DSNs and file content.
- Server-side target driver validation/gating.
- SQLite target connection allowed-source controls.
- Org membership gate before org-scoped RBAC.
- Concrete resource context middleware for cross-resource checks.
- Policy grant guard for owner-level org permissions.
- Auth/session revocation.
- Query cancellation.
- Workspace private/shared file scoping.
- Personal-space feature gate.

Important open gaps:

- Audit trail for policy and data access events.
- Tamper-evident audit logs.
- SSO/SCIM identity lifecycle.
- SSRF-safe cloud deployment model.
- Strong SQL parser for DQL/DML/DDL classification.
- Distributed cache invalidation.
- Binding expiry enforcement.
- Service accounts/API tokens.

SQLWarden is primarily self-hosted. Any future hosted/cloud offering needs stronger controls around SSRF, target network egress, tenant isolation, audit integrity, and managed identity lifecycle before it is safe.

## Desktop/Wails Direction

Future Wails support should reuse `internal/web` instead of importing `cmd/api`.

Recommended model:

- `deployment_mode=desktop` controls runtime packaging and local loopback behavior.
- `access_mode=single_user` controls bootstrap and account behavior.
- Desktop still creates a real local account and local organization.
- Desktop does not bypass RBAC for org-owned resources.
- Personal spaces remain optional sandboxes for multi-user deployments, not the desktop security model.
- Wails-specific code should be isolated behind a small bridge layer in the frontend and a future `cmd/desktop` entrypoint.

Future desktop may support multiple remote SQLWarden backends, such as separate prod and non-prod enterprise instances. That should be modeled as a client-side backend registry, not as a change to the server authorization model.

## Open-Core / Enterprise Direction

The current repository has no `enterprise/` tree. If SQLWarden later adopts an open-core model, keep the core platform in Apache-licensed packages and isolate proprietary add-ons.

Likely enterprise-only features:

- SAML/OIDC/LDAP SSO.
- SCIM provisioning.
- Tamper-evident audit logs.
- SIEM forwarding.
- Advanced compliance packaging.
- License enforcement.
- Air-gapped enterprise packaging.

The core RBAC engine, local auth, SQL IDE, driver layer, and self-hosted server should remain usable in the community distribution.

## Implementation Invariants

Do not violate these without intentionally changing the architecture:

- `POST /api/setup` is self-sealing.
- Org membership is required before org-scoped RBAC access.
- Accounts do not carry permissions directly.
- Instance-admin status does not grant org permissions.
- Resource creation helpers must populate `resource_hierarchy`.
- Connections are leaves under environments in RBAC ancestry.
- Workspace creation must seed workspace builtin roles/policies.
- Workspace creation must create the default environment transactionally.
- Builtin roles are immutable.
- Role bindings are idempotent.
- Direct permission bindings are not part of the current API.
- Environment and connection policy bindings must verify the target resource belongs to the requested workspace.
- `workspace_members` child-resource bindings must verify the child belongs to the same workspace.
- Permission scope validation must run before creating custom roles.
- Backend permission catalog remains the source of truth for scope/resource permission metadata.
- Deleting resources must invalidate ancestry cache when relevant.
- Removing memberships should revoke affected live DB sessions where implemented.
- RBAC cache is process-local; multi-node deployments require an explicit invalidation strategy.
